＃ 背景

CARET は主に次の 4 つのレイテンシを表示できます。

- コールバックのレイテンシー
- 通信遅延 (ノード間)
- ノードのレイテンシー (ノード内)
- パスの遅延 (ノード内およびノー​​ド間)

次の図は、ノード レイテンシとパス レイテンシの定義を示しています。

![Definition node and path latency](../imgs/path_and_node_latency.svg)

コールバック レイテンシはコールバック関数の実行時間として定義され、単純なトレースポイントで測定できます。ノード間の通信待ち時間は、パブリッシュの呼び出しからサブスクリプションのコールバックの呼び出しまでの時間として定義できます。コールバックとトピックを特定するのは難しくないため、それらのレイテンシは簡単に計算されます。
ただし、ノード レイテンシとパス レイテンシを機械的に定義するのは困難です。ノードレイテンシ、つまり特定のノードの入力から出力までの経過時間は、ROS 層では識別できず、その定義はアプリケーション実装のパターンに依存します。パス遅延はノード遅延と通信遅延の組み合わせとして定義され、ノード遅延だけでなく実装パターンにも依存します。パスは、トピック メッセージを介して接続されるノードの組み合わせです。アプリケーション内のパスの数はノードの組み合わせの数に等しいため、複雑で大規模なアプリケーションではパスの数が多くなります。
ノードとパスのレイテンシーを機械的に定義するこのような困難に対処するために、CARET ではユーザーが「**アーキテクチャ オブジェクト**」と呼ばれる構成可能なオブジェクトを介してノードとパスのレイテンシーを手動で定義する必要があります。

## アーキテクチャ オブジェクト

アーキテクチャ オブジェクトには 2 つのセクションがあります。アプリケーションの構造とレイテンシの定義。アプリケーション構造のセクションでは、ターゲット アプリケーションのコンポーネントとその接続について説明します。これらは、エグゼキュータ、ノード、コールバック グループ、コールバック、トピック、タイマーとして表されます。CARETは、[CTF](https://diamon.org/ctf/)ベースの記録データからアプリケーション構造セクションのみを含むアーキテクチャオブジェクトをロードできます。アプリケーション構造セクションは、構造が変更されたり、コンポーネントの名前が変更されたりしない限り、同じままです。

一方、元のアーキテクチャ オブジェクトのレイテンシ定義セクションは、ユーザーが定義を追加する直前には空になっています。ユーザーは、元のアーキテクチャ オブジェクトにノード レイテンシとパス レイテンシの定義を追加することが期待されます。CARET は、ユーザーがノードとパスの遅延の定義を追加するのに役立ちます。
